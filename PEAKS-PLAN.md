This document is the implementation bootstrap for a Garmin Connect IQ device app that labels nearby mountain peaks on a simple horizon/compass line. It replaces the messy pasted planning transcript with a structured build plan while preserving the original notes at the bottom for reference.

## Product Goal

Build a Garmin Connect IQ app that works like a simplified offline PeakFinder:

```text
              Mt Wilson
                  |
      Mt Lukens   |       Strawberry Peak
          |       |              |
W --------|-------|--------------|-------- E
             NE 052 deg   Range 30 mi
```

The app should let the user stand on a trail, rotate their wrist/body, and see peak names in the direction the watch is facing.

The first version should answer:

```text
What named peaks are in the direction I am facing?
```

It should not initially try to prove:

```text
These peaks are definitely visible above intervening terrain.
```

True terrain visibility requires DEM/horizon math and should be treated as a later phase.

## Recommended Connect IQ App Type

Use a Connect IQ **device app**, not a watch face and not a data field.

Reasons:

- A watch face should be passive and low power; this peak labeler is interactive.
- A data field is useful during activities, but UI and control flow are constrained.
- A device app gives the most room for custom views, button controls, manual compass correction, range filters, peak list screens, and settings.

## Data Architecture Decision

Use this architecture for MVP:

```text
Laptop Python generator -> generated Monkey C data file -> build PRG -> deploy PRG to watch
```

Do **not** start by copying arbitrary JSON files to the watch and trying to load them at runtime.

Reasons:

- Connect IQ apps are most reliable when data is compiled into the PRG as source/resources.
- Runtime JSON parsing on the watch wastes heap and CPU.
- Garmin `Application.Storage` is for settings/small state, not a primary regional database.
- Arbitrary side-loaded file access is not a good foundation for a robust private Garmin app.

Keep JSON as a laptop debug/intermediate format, but generate Monkey C arrays for the app.

## Storage Constraints And Practical Limits

Design conservatively.

Use `Application.Storage` only for settings:

```text
range filter
manual compass correction
units
last selected screen
last known location metadata
```

Do not store thousands of peaks in `Application.Storage` for MVP.

Garmin object storage has per-value limits and device-specific total limits. The planning notes mention a 32 KB per-value limit. Even if the app can work around this with chunking, it is unnecessary for the first version.

Initial dataset targets:

```text
MVP:                     200-500 peaks
Comfortable test range:  500-1000 peaks
Stress test later:       1000-2000 peaks
Avoid at first:          10000+ peaks
```

The watch should draw only a small subset per screen update:

```text
max visible labels: 8-12
max tick marks:     maybe 20-30
```

## Peak Data Sources

### Primary MVP Source: OpenStreetMap Overpass

Use Overpass API to query OSM peaks:

```text
natural=peak
name exists
within radius of center lat/lon
```

Advantages:

- No API key.
- Easy to query by radius.
- Often includes local named peaks.
- Often includes elevation via `ele` tag.

Disadvantages:

- Data quality varies.
- Prominence is often missing.
- Public Overpass instances can timeout or rate-limit.

The generator should cache Overpass responses under `cache/` so repeated tuning does not hammer the API.

### Later Source: USGS GNIS

USGS GNIS can provide official named U.S. geographic features. It may be useful as a second source or cross-check.

Do not add GNIS in MVP unless OSM data is clearly inadequate.

### Later Source: Manual Favorites

Support a manually maintained local file later:

```text
data/manual_peaks.csv
```

Use it for important peaks that are missing, badly named, or need custom priority.

## Generator Processing Steps

1. Parse command-line arguments.
2. Convert radius miles to meters.
3. Build Overpass query for named `natural=peak` nodes around center.
4. Load cached response if available; otherwise call Overpass.
5. Parse OSM elements.
6. Require a non-empty `name` tag.
7. Parse `ele` if present. Accept common formats:
   - `5713`
   - `5713 ft`
   - `1741 m`
   - `1,741 m`
8. Normalize elevation to meters and feet.
9. Compute distance from dataset center.
10. Compute bearing from dataset center for debug reports.
11. Apply filters:
    - within radius
    - has name
    - min elevation if provided
    - no obvious duplicate names within a small distance
12. Score/sort peaks by importance.
13. Keep top `--max-peaks`.
14. Write debug JSON.
15. Write `source/PeakData.mc`.
16. Print summary:
    - source count
    - named count
    - after filters count
    - emitted count
    - approximate generated file size
    - top 20 peaks

## Peak Ranking Heuristic

Because prominence is often missing, use a simple priority score for MVP:

```text
score = elevation_weight + distance_weight + name_quality_weight
```

Suggested behavior:

- Prefer named peaks.
- Prefer higher elevation.
- Prefer closer peaks, but do not let tiny nearby bumps crowd out major peaks.
- Prefer manually favorited peaks when manual data exists.
- Optionally deprioritize names that look generic or minor.

A useful first sort:

```text
1. manual priority descending
2. elevation_ft descending
3. distance_mi ascending
4. name ascending
```

Then the watch can further prioritize visible labels by:

```text
1. bearing near current heading
2. distance within selected range
3. priority/elevation
4. label collision avoidance
```

## Generated Monkey C Data Format

Start with simple generated Monkey C arrays.

File:

```text
source/PeakData.mc
```

Shape:

```monkeyc
module PeakData {
    const DATASET_ID = "socal_lcf_60mi";
    const GENERATED_AT = "2026-06-16T12:00:00-07:00";
    const SOURCE = "osm-overpass";
    const CENTER_LAT_E7 = 341990000;
    const CENTER_LON_E7 = -1182000000;
    const RADIUS_MI = 60;

    // Fields: [lat_e7, lon_e7, elev_m, priority, name]
    const PEAKS = [
        [342244000, -1180572000, 1741, 900, "Mt Wilson"],
        [342692000, -1182390000, 1547, 820, "Mt Lukens"]
    ];
}
```

Use integer-scaled coordinates:

```text
lat_e7 = round(latitude * 10000000)
lon_e7 = round(longitude * 10000000)
elev_m = integer meters
```

This avoids floating-point constants in large generated arrays and keeps data compact enough for the first version.

### Later Compact Format

If the simple array is too large, move to string table plus numeric records:

```monkeyc
module PeakData {
    const NAMES = [
        "Mt Wilson",
        "Mt Lukens"
    ];

    // Fields: [lat_e7, lon_e7, elev_m, priority, name_index]
    const PEAKS = [
        [342244000, -1180572000, 1741, 900, 0],
        [342692000, -1182390000, 1547, 820, 1]
    ];
}
```

Do not start with binary unless the generated Monkey C format proves too large.

## Watch Runtime Model

The watch should compute peak positions from live user location and heading.

Do not precompute bearing only from the dataset center as the final runtime bearing. The user may be miles from the dataset center.

For each peak at runtime:

```text
current user lat/lon
peak lat/lon
selected range
current heading
manual heading correction
```

Compute:

```text
distance_m
bearing_deg
relative_bearing_deg
elevation_angle_deg optional later
screen_x
label placement
```

### Update Cadence

Keep runtime cheap.

Suggested cadence:

```text
GPS/location update:      when available or every few seconds
heading update:           every display update if sensor allows
peak filtering:           when location/heading/range changes
drawing:                  onUpdate
```

For the MVP, scanning 500-1000 peaks per update may be acceptable on Epix Pro, but keep the code structured so it can cache computed distances and only recompute bearing when location changes.

### Real Device GPS And Heading State

The watch must not treat startup defaults as real device state.

The current implementation starts with the generated dataset center as a demo location and a fixed heading so the app can run in the simulator. On the real Epix Pro, those defaults must be treated only as fallback values. The UI should clearly show whether the app is using real watch data or fallback data.

Track GPS and heading separately:

```text
GPS state:
  DEMO      using generated dataset center because no watch position is available
  WAIT      location events are enabled, but no usable fix has arrived
  POOR      a position exists, but quality/accuracy is too weak for confident labels
  USABLE    a live position is good enough to compute peak bearings
  GOOD      a live position is strong enough for normal outdoor use

Heading state:
  WAIT      heading events are enabled, but no heading has arrived
  LIVE      heading is arriving and is being used to rotate the peak window
  STALE     a heading arrived before, but it has not updated recently
  LOCK      user locked heading with SELECT
  MANUAL    user is controlling heading with UP/DOWN fallback controls
```

Important behavior:

- Do not show `GPS location` just because one coordinate was received; prefer a quality/accuracy-aware label.
- Do not continue saying `Demo location` after a usable live GPS position is accepted.
- If heading updates only when the watch is tilted, surface that as a heading availability/calibration problem instead of hiding it.
- Peaks should move left/right when `_heading` changes because relative bearing is computed during drawing.
- Peaks should be recomputed from live `_userLat` and `_userLon` once GPS is usable; the generated dataset center should only remain as the pre-GPS fallback.
- Manual heading mode remains useful for simulator testing, indoor testing, and devices/situations where compass heading is unavailable.

## Math Requirements

Create a small math module:

```text
source/PeakMath.mc
```

Functions:

```text
degToRad(deg)
radToDeg(rad)
normalize360(deg)
normalize180(deg)
distanceMeters(lat1Deg, lon1Deg, lat2Deg, lon2Deg)
bearingDegrees(lat1Deg, lon1Deg, lat2Deg, lon2Deg)
relativeBearing(peakBearing, heading, correction)
elevationAngleDegrees(userElevM, peakElevM, distanceM)
```

Use haversine or equirectangular distance.

For local 100-mile radius, equirectangular is fast and acceptable:

```text
x = delta_lon_rad * cos(avg_lat_rad)
y = delta_lat_rad
distance = earth_radius_m * sqrt(x*x + y*y)
```

For bearing, use the standard great-circle initial bearing formula:

```text
bearing = atan2(
  sin(delta_lon) * cos(lat2),
  cos(lat1) * sin(lat2) - sin(lat1) * cos(lat2) * cos(delta_lon)
)
```

Normalize to `0..360`.

Relative bearing:

```text
relative = normalize180(peakBearing - heading - manualCorrection)
```

Screen position:

```text
screenX = centerX + (relative / halfWindowDeg) * usableHalfWidth
```

Only draw if:

```text
abs(relative) <= halfWindowDeg
```

### Label Collision Strategy

Do not draw every peak label.

MVP strategy:

1. Compute candidate peaks in current bearing window and range.
2. Sort by priority:
   - closer to center heading
   - higher priority/elevation
   - closer distance
3. Place labels one by one into 2-3 rows.
4. If label bounding box overlaps existing label, try another row.
5. If all rows overlap, draw only a tick or skip label.
6. Limit labels to 8-12.

Keep tick marks for additional peaks optional. Too many ticks can become clutter.

## PeakFinder Reference And Scope Boundaries

This project is inspired by PeakFinder:

```text
https://www.peakfinder.com/mobile/
```

PeakFinder is the benchmark user experience, but the Garmin app should not try to clone all of PeakFinder in the first version.

PeakFinder capabilities worth understanding:

- worldwide offline mountain database with more than 1,000,000 peaks;
- integrated elevation model for rendering 360 degree panoramic mountain silhouettes;
- current-location panorama generation;
- camera overlay mode;
- compass calibration and manual alignment correction;
- photo snapshots and later photo editing/export;
- touch a mountain label for more information;
- telescope/zoom mode to identify less prominent peaks;
- vertical fly-up/fly-down to reveal what is hidden behind ridges;
- sliders for compass correction, visibility range, and sun/moon date/time;
- marks/favorites synced through an account or stored locally;
- arbitrary viewpoint selection;
- visible-peaks list;
- downloadable offline regions.

For this Garmin project, treat PeakFinder as inspiration for interaction concepts and long-term direction, not as the MVP feature set.

### Garmin MVP Scope Compared With PeakFinder

Version 1 generated Monkey C

Shape:

Fields:

[lat_e7, lon_e7, elev_m, prominence_m, name]

Pros:

Very easy.
Debuggable.
No parser.
No separate file copy.
Works offline.

Cons:

Rebuilding app required to change dataset.
Large arrays may eventually hit app/resource size limits.
Strings are not as compact as binary.
Version 2 compact source

Use string table and numeric records:

const NAMES = [
  "Mt Wilson",
  "Mt Lukens"
];

  [342244000, -1180572000, 1741, 0, 0],
  [342692000, -1182390000, 1547, 0, 1]
];

Fields:

[lat_e7, lon_e7, elev_m, prominence_m, name_index]

This reduces duplicated strings and prepares you for binary.

Use a packed binary file generated into resources, or a byte array.

Record concept:

20. Final design decision

The best design is:

Laptop does heavy data gathering.
Laptop writes a compact generated Monkey C data file.
Garmin app contains the data at build time.
Watch runs fully offline.
Watch computes bearings from live GPS and compass.
Watch draws a simple peak-label line.

Your user workflow becomes:

.\deploy-peakline.ps1 -Lat 34.199 -Lon -118.200 -RadiusMi 60

Behind the scenes:

generate peaks â†’ write PeakData.mc â†’ build PRG â†’ copy PRG to watch

That avoids the biggest Garmin headaches: large runtime JSON parsing, arbitrary side-loaded files, and object-store limits.

So yes: donâ€™t start with JSON on the watch. Start with JSON on the laptop, generated Monkey C on the watch, and only move to binary if the compiled app gets too big or slow.
